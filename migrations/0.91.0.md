## ADM Validation Goes Per-Album

Issue [#353](https://github.com/bitwize-music-studio/claude-ai-music-skills/issues/353). The April 2026 ADM pipeline build-out (#347, #348, #349, #350, #351, #352) revealed that ADM is an Apple-submission-tier niche that rarely matters for Suno workflows. Rather than rip it out, this release flips the default so it runs only on albums that explicitly opt in.

### What changed for me as a user?

**If you never set `adm_validation_enabled` in global config**: nothing changes. ADM was off, ADM stays off.

**If you had `mastering.adm_validation_enabled: true` in `~/.bitwize-music/config.yaml`**: that line is now ignored. ADM no longer runs on any of your albums. To restore the previous behavior on a specific album, add the per-album frontmatter block:

```yaml
mastering:
  adm_validation_enabled: true
```

Inside the album README's existing frontmatter block (the YAML at the top of the file).

**If you submit to Apple Music as Apple Digital Masters**: keep ADM enabled per album that you submit. Flip it on in the album's frontmatter; leave global config alone.

### Why per-album?

ADM is a submission-tier decision that lives with the album, not with the operator. An album either ships to Apple Music as ADM or it doesn't — that's a property of the release, not of who happens to be at the keyboard. Putting the flag in the album README means the decision travels with the album across machines, contributors, and re-master cycles.

### Companion architecture (kept, not removed)

- Per-track ADM ceiling tightening (#348)
- Dark-casualty classification (`is_dark` on `analyze_track`, `band_energy['high_mid'] < 10%`)
- Harmonic excitation in polish for dark-material stems (#351)
- Warn-fallback contract for non-tightenable failures
- Post-master spectral regression guard (#349)
- `ADM_VALIDATION.md` sidecar with `dark_casualties`, `tightened_tracks`, per-track final ceilings

These all continue to apply when ADM is opted in. None of them are user-facing knobs in normal use.
